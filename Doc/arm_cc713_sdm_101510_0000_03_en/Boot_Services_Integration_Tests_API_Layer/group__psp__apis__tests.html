Detailed Description

Contains macros to validate PSP integration.

The following flow describes the production secure provisioning test flow.

The production secure provisioning test

Note
Production secure provisioning is only relevant for CM or DM LCS, and therefor only tested in these LCS.
Before the flow begins, you must load the secure package into a DMA-able address.
  1. To initialize the CryptoCell boot sequence, call CC_BsvInit().
  2. To return the current life cycle, call CC_BsvGetAndInitLcs().
  3. To read chip indication and verify that the system is in the required chip indication (test or production), call CC_BsvChipIndicationGet().
  4. If the LCS is CM or DM (the only relevant LCS for this flow):
    1. Set CryptoCell to secure provisioning state, call CC_BsvSecureProvEnableSet(). After a successful completion of this function the system performs a full reset, and the boot sequence starts from the main entry point again.
    2. Initialize a full start of the boot sequence, call CC_BsvInit() and CC_BsvGetAndInitLcs(). Calling CC_BsvInit() validates that the reset was completed and the system is in steady state. Calling CC_BsvGetAndInitLcs re-reads LCS and re-validates the life cycle.
    3. Read the secure production indication to verify that CryptoCell is in the secure production state, call CC_BsvSecureProvEnableGet().
    4. If secure production is enabled, proceed with the secure package verification, call CC_BsvProductionPkgVerify(). The function does the following:
      • Verifies the secure package validity and authenticity (the secure package contains the encrypted and authenticated production tool).
      • Returns the address of the production application to branch to.
      Note
      In our flow we did not provide the implementation of the branching to the utility, as it is dependent on the CPU architecture.
  5. After the code branches to the production application the ICV/OEM assets are decrypted and burnt to the OTP. The system must be reset to move to the next LCS.

You must replace this macro with an implementation that is customized for your system.

◆ USER_JUMP_TO_ADDR

#define USER_JUMP_TO_ADDR (   addr)
Value:
{ \
return TE_RET_JUMP_TO_ADDR; \
}

This macro simulates jump to address functionality. The address to jump to is the returned address from CC_BsvProductionPkgVerify API. This address is the location of the decrypted DMPU/CMPU applications.

Note
You must replace this macro with an implementation that is customized for your system.
